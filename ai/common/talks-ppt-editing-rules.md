# talks/ PPT 修改规则

本规则适用于所有 AI 工具（Claude / Codex / Antigravity 等）在本仓库中讨论或修改 `talks/` 目录下的答辩、汇报或演示文稿。

## 核心流程

1. **先方案、后修改**：每次讨论 PPT 修改时，必须先给出拟修改方案，包括修改哪一帧、改什么内容、版式如何调整、是否影响相邻帧。待用户确认后，才能修改 `.tex` 文件。
2. **修改后必须编译**：确认执行后，应修改对应 `.tex` 文件，并编译生成相应 `.pdf`，方便用户审查；必要时渲染目标页截图进行版式检查。
3. **只改指定帧**：修改 `.tex` 文件时，只能编辑用户指定帧或明确相关的极小范围代码。严禁进行大范围模糊匹配、整段覆盖、跨帧批量替换或无关重排，以免破坏前后帧内容与排版。
4. **保持风格统一**：修改某一帧时，应同步参考与其功能相近的帧（如同一 Part、同类流程图、同类结果表格、同类 takeaway）的排版风格，尽量统一标题层级、留白、字号、线条、箭头、表格和底部总结口径。
5. **保护既有成果**：若发现 `.tex` 与已生成 `.pdf` 不一致，或疑似被其他工具误删/覆盖，应先报告差异并给出恢复方案，不能直接用旧内容覆盖当前文件。

## 执行边界

- 若用户只是在讨论“如何修改”，不要直接动文件。
- 若用户明确说“按建议修改 / 更新 PPT / 编译 PDF”，才进入文件修改与编译步骤。
- 若需要跨多帧统一风格，必须说明涉及哪些帧，并优先采用局部、可审查的小改动。
- 若修改可能影响页码、帧数、版式模板或全局宏，应先单独说明风险，等待确认。

## 编译与 QA 规则

- 每次修改 `talks/` 下的 `.tex` 后，必须重新 XeLaTeX 编译正式 PDF。
- 编译后必须检查 Overfull 数量，明确说明“无新增 Overfull”或说明新增项来源。
- 对用户关心的页面，优先渲染目标页截图进行视觉 QA。
- 若本轮只修改 guide / status / 规则文档，没有改 PPT 源文件，可不编译，但最终回复中必须说明“本轮未改 PPT，所以未重新编译”。
- 不要把已有 Overfull 当成新问题；以修改前后数量是否增加作为当前任务的 QA 边界。

## 版式一致性规则

- 同一 Part 内带蓝色小标题与横线的页面，应统一标题区间距：小标题与顶部段落距离略增，小标题到下方横线距离收紧。
- 调整某页标题区间距时，应检查左右栏横线是否在同一水平线上。
- 底部参考文献应尽量规范、简短、统一；长解释放入对应 guide，不压进 PPT 脚注。
- 主帧文字只保留面向评委的结论和口径；事实边界、来源背景、答辩追问放入 guide。
- 不要把“技术任务”和“平台支撑/能力来源”混写：技术任务写在工作计划或方法体系，平台支撑说明团队和资源凭什么能支撑。

## Guide 同步规则

- 每个已定稿或准定稿的核心页，应有对应 guide 解释 PPT 中每一个术语、公式、数值、图示、参考文献和诚实边界。
- 改 PPT 可见文字时，应同步更新对应 guide 中的讲稿、Q&A 和事实边界。
- 改 guide 口径但不改 PPT 时，应在 `ai/common/status.md` 记录这一点，避免后续误以为 PPT 已改。
- 数值事实不能凭聊天记忆写入 PPT；帧 7/8/10 的数值应回到对应 pipeline results 文件核对。
