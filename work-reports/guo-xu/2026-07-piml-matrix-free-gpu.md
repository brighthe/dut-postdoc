## 四、三条技术线的已有基础与下一步

### 4.1 PIML：局部子结构前向链路

> **长期指导文档**：[[../../research/technical-lines/piml-research-guide]]。

#### 已完成

- 跑通“局部细尺度密度 → 子结构静力缩聚 → 等效刚度 $K_s^j$ → 全局接口缩聚方程 → 接口求解/细尺度恢复”的基本链路。
- 精确缩聚与全尺度 Schur 补误差为 $10^{-15}$ 量级；接口解与全尺度直解保持 $10^{-14}$–$10^{-12}$ 量级一致。
- 训练极小 MLP 直接预测 $K_s$，部署算例上逐子结构相对预测误差均值为：
  - $5\times5$：$1.6\times10^{-3}$；
  - $10\times10$：$8.2\times10^{-3}$。
- 两档均优于只用局部平均密度的 Mock 对照。

> **汇报边界**：`1.6e-3 / 8.2e-3` 是局部 $K_s$ 预测误差，不是位移、柔顺度、灵敏度或最终拓扑误差；当前尚未完成结构保持参数化、多尺度灵敏度和完整拓扑优化闭环。

#### 后续工作

- 研究对称性、正定性和能量一致性可控的结构保持参数化。
- 建立局部 $K_s$ 预测误差向全局位移、柔顺度、灵敏度和求解器收敛的传播评价。
- 补齐多尺度灵敏度分析，将前向原型接入完整拓扑优化闭环。
- 围绕同一局部等效算子任务开展模型选型 benchmark，比较预测误差、结构保持、全局响应和部署成本。

### 4.2 Matrix-Free：全局无矩阵状态方程基础

> **长期指导文档**：[[../../research/technical-lines/matrix-free-research-guide]]。

#### 已完成

- 在积分点直接做 $B^TDBx$ contraction，不形成全局 $K$，也不预先形成单元 $K_e$。
- Matrix-Free MatVec 与显式矩阵乘的相对误差达到 $10^{-15}$–$10^{-13}$。
- Matrix-Free CG 已跑通小规模 $KU=F$ 求解闭环，残差约 $10^{-11}$–$10^{-10}$。
- NumPy、PyTorch CPU、CUDA 三后端结果一致。

> **汇报边界**：当前局部算子仍是常规有限元单元作用，尚未正式换成 PIML 子结构 $K_s^j$；已跑通的是小规模状态方程原型，不能表述为大规模融合求解器已经完成。

#### 后续工作

- 先接入精确 $K_s$，建立子结构级 Matrix-Free MatVec 与显式缩聚矩阵乘的机器精度对照。
- 建立 Krylov 与 none/Jacobi/block Jacobi 等基础预条件配置的收敛和性能基线。
- 推进 PETSc Shell Matrix 的算法层集成，使无矩阵算子能够进入现有求解流程。
- 扩展代表性三维和大规模算例，系统记录残差、迭代数、时间和内存。

### 4.3 GPU/HPC：性能证据与工程基础

> **长期指导文档**：[[../../research/technical-lines/gpu-hpc-research-guide]]。

#### 已完成

- `soptx` 原型在 13.2 万 DOF 下，单次 GPU MatVec 约 $11.9\times$，内存估计从 42.1 MB 降至 4.0 MB。
- 本人编写的 `mfleo` PA / Matrix-Free 包已完成 650 万 DOF 的 GPU/MPI 端到端 CG；1–32 个 MPI 进程配置下，相对同规模 `MFEM PA` 基线约 $3.72\times$–$12.74\times$。
- `mfleo` 已完成 P2 tet 的 Jacobi、Chebyshev 预条件子测试；CPU 约 $1.20\times$–$1.21\times$，GPU 多数配置约 $4\times+$。

> **汇报边界**：$11.9\times$ 是 `soptx` 单次 MatVec 加速，不是完整 solve 加速；`mfleo` 结果证明已有 GPU/MPI、Krylov 和基础预条件子工程经验，但不是当前 soptx/PIML 一体化结果。

#### 后续工作

- 实现 PIML 批量推理、局部算子作用和 scatter-add 的 GPU 批处理管线。
- 开展端到端 solve profiling，分解模型推理、局部作用、数据搬移、通信和 Krylov 迭代成本。
- 评估显存占用、吞吐和问题规模之间的关系，避免只报告单次算子加速。
- 在单卡原型稳定后推进多后端、多节点和软件模块集成。

## 五、三条技术线融合的核心问题

| 融合问题 | 涉及技术线 | 当前需要回答的问题 |
|---|---|---|
| 局部—全局算子接口 | PIML × Matrix-Free | $K_s$ 的自由度顺序、批次布局、边界处理、局部作用和 scatter-add 如何形成统一接口？ |
| 结构保持与误差传播 | PIML × Matrix-Free | 预测算子的对称性、正定性和能量一致性如何保证，局部误差如何传播到全局响应？ |
| Krylov 收敛与预条件 | PIML × Matrix-Free × GPU | 预测算子的谱扰动是否破坏 CG/GMRES 收敛，基础及可扩展预条件子如何设计？ |
| 端到端性能 | PIML × Matrix-Free × GPU | 模型推理、局部算子、scatter-add、数据搬移和通信的总成本能否形成完整 solve 收益？ |
| 优化闭环与软件集成 | PIML × Matrix-Free × GPU | 如何补齐灵敏度、拓扑优化闭环和可复用软件接口，并保持结果可验证？ |

核心问题可以压成一句：

> 现在不是三块都从零开始，而是各自已有基础；下一步的科学和工程增量，是让 PIML 预测的局部算子在保持数值结构的前提下，可靠进入全局 Matrix-Free Krylov 求解，并在 GPU 上形成端到端收益。

## 六、融合实施路线与首批交付结果

### 6.1 建议顺序

```text
研究院求解器与性能基线
  -> 精确 K_s 的子结构级全局 Matrix-Free
  -> Krylov + 基础预条件子
  -> 替换为 PIML 预测 K_s，检查结构与误差传播
  -> PIML 推理 + 局部算子 GPU 批处理
  -> 灵敏度、优化闭环、多节点和软件集成
```

### 6.2 为什么按这个顺序

- 先用精确 $K_s$，可以隔离 Matrix-Free 接口错误与 PIML 预测误差。
- 先建立 Krylov/预条件基线，才能判断预测算子是否破坏收敛。
- 先稳定数据接口和正确性，再做 GPU，性能数字才可解释。
- 灵敏度和优化闭环放在接口可靠之后，避免同时引入过多误差源。

### 6.3 第一批可交付结果

1. 研究院 Hypre/PETSc KSP 代表算例的收敛性与性能基线。
2. 精确 $K_s$ 下，子结构级 Matrix-Free MatVec 与显式缩聚矩阵乘的机器精度对照。
3. none/Jacobi/block Jacobi 等基础配置的残差、迭代数、时间和内存表。
4. 换入 PIML $\widehat K_s$ 后的对称性、最小特征值、谱性质、全局位移误差和 Krylov 收敛报告。
5. 批量 PIML 推理、局部算子作用和 scatter-add 的 GPU profiling。
